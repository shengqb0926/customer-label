## 🔬 四、根因分析

### 4.1 直接原因

```
【技术层面】
[详细描述直接导致故障的技术原因，例如:

数据库连接池配置不合理:
- max_connections = 100
- 实际并发峰值达到 150+
- 新请求无法获取连接，超时失败
- 应用层未设置合理的连接超时和重试机制]

【代码/配置层面】
[具体到代码或配置的问题，例如:

application.yml 中配置:
spring:
  datasource:
    hikari:
      maximum-pool-size: 100  # ❌ 配置过小
      connection-timeout: 30s  # ⚠️ 超时时间过长
      
应调整为:
      maximum-pool-size: 200  # ✅ 根据压测结果调整
      connection-timeout: 5s   # ✅ 快速失败]
```

### 4.2 间接原因

```
【流程层面】
- [ ] 容量规划不足：未根据业务增长及时调整资源配置
- [ ] 监控告警缺失：连接池使用率无预警机制
- [ ] 应急预案缺失：无连接池耗尽的处理 Runbook
- [ ] 变更管理不当：配置修改未经过充分测试

【管理层面】
- [ ] 人员培训不足：值班人员对数据库连接管理不熟悉
- [ ] 文档不完善：缺少数据库配置最佳实践文档
- [ ] 工具链缺失：缺乏自动化容量评估工具
```

### 4.3 5 Why 分析法

```
Why 1: 为什么服务不可用？
→ 因为应用无法获取数据库连接

Why 2: 为什么无法获取数据库连接？
→ 因为连接池已耗尽（100 个连接全部占用）

Why 3: 为什么连接池会耗尽？
→ 因为并发请求激增（促销活动），且部分连接未及时释放

Why 4: 为什么连接未及时释放？
→ 因为存在慢查询（平均执行时间 5s+），占用连接时间过长

Why 5: 为什么存在慢查询？
→ 因为新增的统计查询未添加合适索引，全表扫描

根本原因:
1. 容量规划未考虑业务峰值场景
2. 慢查询监控和治理机制缺失
3. SQL 审查流程执行不到位
```

## ✅ 五、处置过程

### 5.1 临时措施

**措施 1: 紧急扩容连接池**
```bash
# 执行时间：HH:MM+20
# 执行人：[姓名]

# 1. 修改配置
kubectl set env deployment/customer-label \
  DB_MAX_CONNECTIONS=200

# 2. 重启应用
kubectl rollout restart deployment/customer-label

# 3. 验证恢复
kubectl get pods
curl http://localhost:3000/health
```

**效果验证**:
- ✅ 连接池使用率从 100% 降至 60%
- ✅ API 错误率从 80% 降至<1%
- ✅ 服务在 10 分钟内恢复

---

### 5.2 永久措施

**措施 1: 优化慢查询**
```sql
-- 添加索引
CREATE INDEX idx_customers_level_assets 
ON customers(level, total_assets DESC);

-- 优化查询语句
SELECT /*+ IndexScan(customers idx_level_assets) */ *
FROM customers
WHERE level = 'GOLD' AND total_assets >= 5000000;
```

**措施 2: 调整连接池配置**
```yaml
# application.prod.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 200    # 从 100 调整到 200
      minimum-idle: 20          # 新增
      connection-timeout: 5000  # 从 30s 降到 5s
      idle-timeout: 300000      # 新增
      max-lifetime: 600000      # 新增
```

**措施 3: 增加监控告警**
```yaml
# alerting_rules.yml
- alert: DatabaseConnectionPoolHigh
  expr: pg_stat_activity_count / current_setting('max_connections')::int > 0.7
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "数据库连接池使用率超过 70%"
```
