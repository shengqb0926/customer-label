# OpenSpec 示例变更演示

本目录展示了如何使用 OpenSpec 规范来管理一个完整的功能开发项目。

## 📁 示例变更概览

**变更名称**: `add-smart-tag-recommendation`  
**功能描述**: 客户标签智能推荐系统  
**优先级**: High  
**预估工时**: 120 小时（3 周）  
**创建日期**: 2026-03-26

```
openspec/changes/add-smart-tag-recommendation/
├── proposal.md          # 变更提案

```

### 1. proposal.md - 变更提案

**回答的问题**:
- ✅ 为什么要做这个功能？（背景和目标）
- ✅ 要做什么？（技术方案概述）
- ✅ 影响哪些模块？（影响范围）
- ✅ 如何做？（技术选型）
- ✅ 有什么风险？（回滚方案、安全性评估）

**核心内容**:
- 问题陈述：当前标签管理系统效率低、质量差、重复率高
- 解决方案：智能标签推荐系统（规则引擎 + 聚类引擎 + 关联引擎）
- 技术选型：Node.js + TypeScript + NestJS + TensorFlow.js
- 影响范围：新增推荐模块、修改标签管理模块、数据库变更
- 验收标准：功能、性能、质量、业务四大维度

**使用场景**:
- 向团队展示新功能想法
- 获取项目干系人认可
- 评估技术可行性

---

### 2. spec.md - 功能规范

**回答的问题**:
- ✅ 具体实现哪些功能？（功能需求）
- ✅ 功能的行为是什么？（Given/When/Then 格式）
- ✅ 接口如何定义？（TypeScript 类型）
- ✅ 数据模型如何？（数据库表结构）
- ✅ 如何验收？（验收标准）

**核心内容**:
- **功能需求**:
  - 客户数据分析（画像构建、特征提取）
  - 标签推荐引擎（规则、聚类、关联三种方式）
  - 标签质量评分（覆盖率、区分度、稳定性、业务价值）
  - 冲突检测（命名、逻辑、冗余三类冲突）
  - 反馈机制（用户反馈收集和分析）

- **非功能需求**:
  - 性能要求：响应时间 < 500ms，并发 100 req/s
  - 可靠性：可用性 > 99.5%
  - 安全性：JWT 认证、RBAC 权限、审计日志

- **接口定义**: RESTful API 端点、请求/响应格式

- **数据模型**: 5 个核心表的详细设计

**使用场景**:
- AI 编程助手读取并实现功能
- 开发人员理解需求细节
- 测试人员编写测试用例
- 产品验收功能

---

### 3. design.md - 技术设计

**回答的问题**:
- ✅ 系统架构如何？（架构图、分层设计）
- ✅ 模块如何划分？（模块职责、依赖关系）
- ✅ 核心组件如何实现？（类设计、算法流程）
- ✅ 数据库如何设计？（ER 图、索引策略）
- ✅ 如何部署和监控？（容器化、监控指标）

**核心内容**:
- **架构设计**:
  - 系统架构图（客户端层 → API 网关 → 应用服务 → 业务逻辑 → 数据存储）
  - 技术栈选型（NestJS、TypeORM、Redis、Bull）
  - 模块划分（8 个核心模块）

- **核心组件设计**:
  - RuleEngine（规则引擎）：加载规则、评估规则、生成推荐
  - ClusteringEngine（聚类引擎）：特征工程、K-Means 算法、簇分析
  - AssociationEngine（关联引擎）：Apriori 算法、规则挖掘
  - ScoringService（评分服务）：四个维度评分器

- **数据库设计**:
  - ER 图展示表关系
  - 分区策略优化性能

- **部署架构**:
  - Docker Compose 配置
  - Prometheus + Grafana 监控
  - 告警规则配置

**使用场景**:
- 开发人员理解系统架构
- 代码实现的蓝图
- 技术评审材料

---

### 4. tasks.md - 任务列表

**回答的问题**:
- ✅ 需要完成哪些具体任务？（WBS 分解）
- ✅ 每个任务的预估工时？（时间估算）
- ✅ 任务的依赖关系？（执行顺序）
- ✅ 如何跟踪进度？（里程碑、验收标准）

**核心内容**:
- **5 个 Phase**:
  - Phase 1: 基础架构搭建（24 小时）
  - Phase 2: 核心功能实现（56 小时）
  - Phase 3: API 和前端集成（24 小时）
  - Phase 4: 测试和优化（16 小时）
  - Phase 5: 部署和上线（8 小时）

- **任务分解**:
  - 每个 Phase 包含多个 Task
  - 每个 Task 分解为多个子任务
  - 每个子任务有明确的验收标准

- **里程碑**:
  - Milestone 1: 基础架构完成（Day 3）
  - Milestone 2: 核心功能完成（Day 14）
  - Milestone 3: API 和前端完成（Day 17）
  - Milestone 4: 测试完成（Day 19）
  - Milestone 5: 上线准备完成（Day 21）

- **进度跟踪**:
  - 每周的实际完成情况记录
  - 风险和应对措施

**使用场景**:
- 开发人员领取任务
- 项目经理跟踪进度
- 每日站会同步状态

## 💡 最佳实践建议

### ✅ 推荐做法

1. **先写规范后写代码**
   - 确保所有人对需求理解一致
   - 减少返工和沟通成本

2. **保持文档更新**
   - 需求变更时及时更新文档
   - 实现细节变化也要反映到文档

3. **使用 Given/When/Then 格式**
   - 清晰描述功能行为
   - 便于编写测试用例

4. **任务粒度适中**
   - 每个子任务 2-8 小时
   - 便于跟踪和估算

5. **充分利用 AI 工具**
   - 让 AI 读取规范后自动生成代码
   - AI 可以帮助完善文档

### ❌ 避免做法

1. **跳过规范直接写代码**
   - 容易导致需求理解偏差
   - 增加后期返工风险

2. **文档过于简略**
   - AI 无法准确理解需求
   - 开发人员需要反复确认

3. **任务和实现脱节**
   - 任务完成后不及时更新状态
   - 导致进度不透明

4. **忽视验收标准**
   - 没有明确的完成标准
   - 导致质量不可控
